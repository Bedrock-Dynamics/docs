---
title: "What is Substrate?"
description: "Substrate is a native desktop IDE purpose-built for robotics simulation, flight review, and autonomous systems development."
---

Substrate is a desktop development environment for robotics engineers. It combines a full code editor, physics simulation management, 3D visualization, and telemetry analysis into a single native application. Instead of switching between a text editor, terminal windows, RViz, PlotJuggler, and QGroundControl, you work in one integrated workspace.

Substrate is designed for teams building with PX4, ArduPilot, ROS 2, and Gazebo. It runs locally on your machine with GPU-accelerated rendering and connects to both simulated and real hardware over MAVLink or ROS 2 topics.

## The Workflow

A typical development cycle in Substrate follows four stages:

<Steps>
  <Step title="Write">
    Edit flight controllers, ROS 2 nodes, or mission scripts in the built-in code editor. Language Server Protocol support provides autocompletion, diagnostics, and go-to-definition across Python, C++, Rust, and more. Tree-sitter powers syntax highlighting for instant feedback as you type.
  </Step>
  <Step title="Simulate">
    Launch a PX4 SITL or Gazebo simulation directly from the editor. Substrate manages the Docker containers, monitors readiness indicators, and tells you when the simulated vehicle is ready to fly. No terminal juggling required.
  </Step>
  <Step title="Visualize">
    Inspect the simulation in real time through the 3D viewer. View URDF and SDF models, TF transform trees, sensor data overlays, and collision geometry. Telemetry plots update live alongside the 3D scene.
  </Step>
  <Step title="Deploy">
    Connect to real hardware over MAVLink or ROS 2. The same visualization and telemetry tools work with physical vehicles, so you can validate behavior on real systems using the same interface you used in simulation.
  </Step>
</Steps>

## Built Different

Substrate is not an Electron wrapper around web technologies. It is written in Rust with GPU-accelerated rendering via wgpu, giving it native performance characteristics:

- **Zero-latency editing.** The text buffer and input handling run in the UI process on your local machine. Keystrokes never cross a network boundary for basic editing, even when the workspace proxy runs remotely.
- **GPU-accelerated rendering.** The 3D viewer, text rendering, and UI compositor all run on the GPU through Vulkan, Metal, or DirectX backends. Complex scenes with thousands of entities render at full frame rate.
- **UI/Proxy architecture.** Heavy operations like LSP communication, file I/O, Docker management, and simulation streaming run in a separate proxy process. The UI thread is never blocked by disk or network activity.
- **Native memory footprint.** No JavaScript runtime, no bundled Chromium. Substrate typically uses a fraction of the memory required by Electron-based editors.
